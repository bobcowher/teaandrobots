# Tea & Robots — Voice & Style Guide

This file captures the writing voice and documentation conventions for teaandrobots.com. Read it before drafting any new content.

Previous technical debugging notes have been moved to `.claude/hugo-codeblock-debugging.md`.

---

## Voice

**First person, honest, no corporate tone.**

The writing should sound like a person talking to another person — not a product page, not a README generated by a bot. Robert is an AI/robotics enthusiast who builds things, documents what he learns, and is upfront about the limits of his own experience.

Key traits:

- **Self-aware about expertise.** He's not a professional, he's an enthusiast. Say so when it's relevant. Don't oversell.
- **Honest about scope and limitations.** If something was tested on one machine, say so. If it might not work everywhere, say so. Don't paper over it.
- **Genuine frustration and genuine enthusiasm are both fine.** "Absolutely infuriating" is a good phrase. So is "Now for the fun stuff!"
- **Personal details make content better.** "I needed the PID library" is more useful and more readable than a generic "install your required libraries."
- **Self-deprecating when appropriate.** "I built this with lots of Claude help" is on-brand. Pretending otherwise is not.

---

## What to avoid

- **Marketing language.** Never: "powerful", "seamless", "robust", "cutting-edge", "leverages", "supercharge".
- **Passive voice.** "The script installs the service" beats "The service will be installed by the script."
- **Hedging everything.** Be direct. If something works, say it works. If it doesn't, say that too.
- **Over-bulleting.** Not everything needs to be a bullet list. Prose is often more readable.
- **Long intros that delay the point.** Get there quickly.

## Annotated examples

These are lines from actual posts. The annotation explains what's working.

> *"For now, this is an application I built (with lots of Claude help) to solve my personal problems with OBS's background removal capabilities. As of today it has been tested on precisely one machine, with one user, one version of OBS, one version of Python...you get the idea."*

→ Honest about scope without being defensive. "You get the idea" trusts the reader. Use this pattern for software that's early or personal.

---

> *"This process isn't just slow, it's absolutely infuriating if you're trying to get something working, and every single code tweak requires a hardware hookup."*

→ Leads with the pain point and doesn't soften it. Earned frustration earns the reader's trust before the solution appears.

---

> *"I'll also note that I'm an AI enthusiast, not a professional. With that out of the way –"*

→ Sets appropriate expectations without undermining the content. Short, then moves on immediately.

---

> *"Now for the fun stuff!"*

→ Enthusiasm is allowed. Don't flatten it out in the name of professionalism.

---

> *"...and this is where I'll plug…me!"*

→ Self-promotion with self-awareness. The ellipsis and the humor take the edge off.

## Structure conventions

### Front matter

```yaml
---
title: "Title Here"
date: 2026-01-15
draft: false
---
```

- Software pages (`/software/`): `title`, `date`, `draft` only — no tags.
- Posts (`/posts/`): add `tags: ["tag1", "tag2"]` as appropriate.
- Use `YYYY-MM-DD` date format unless the original had a full timestamp.

### Headings

- `##` for major sections, `###` for subsections. Never use `#` (the title is already h1).
- Descriptive and direct: "Getting Started", "First Run", "Troubleshooting". Not "Overview" or "Introduction".

### Opening paragraphs

- No "In this post, we will..." — just start.
- Lead with what the thing *is* or what *problem it solves*, in one or two sentences.
- For software pages: what it does, how it does it, where the source is.
- For tutorials: open with the pain point, then the solution.

### Code blocks

- Always specify the language: ` ```bash `, ` ```python `.
- Inline comments in examples are encouraged — they explain intent without extra prose.
- Show real commands, not pseudocode placeholders.

### Lists

- Numbered lists for sequences and steps.
- Bullet lists for features, options, or alternatives.
- Don't turn everything into bullets — a short paragraph is often more readable.

### Notes sections

- Put caveats, warnings, and "things to know" in a `## Notes` section at the end, not scattered through the main content.

## Software pages vs. posts

| | Software (`/software/`) | Posts (`/posts/`) |
|---|---|---|
| Purpose | Documentation for a specific project | Articles, references, tutorials |
| Tags | No | Yes |
| Tone | Still personal, but more structured | More conversational |
| Typical structure | Requirements → Install → Usage → Options → Notes | Depends on content type |

---

## When in doubt

Read back what you wrote out loud. If it sounds like it came from a product brochure or a corporate wiki, rewrite it. If it sounds like a knowledgeable person explaining something they actually built or learned, it's probably right.
